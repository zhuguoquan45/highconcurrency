# Go GC 机制

## 算法：三色标记 + 混合写屏障

Go 使用**并发三色标记清除**算法，目标是低延迟（STW < 1ms）。

---

## 三色标记

| 颜色 | 含义 |
|------|------|
| 白色 | 未被扫描，GC 结束后白色对象被回收 |
| 灰色 | 已发现但子对象未扫描完 |
| 黑色 | 已扫描完，子对象也已扫描 |

**流程：**
1. STW（Stop The World）：初始标记，将 root 对象（全局变量、栈变量）标为灰色
2. 并发标记：goroutine 和 GC 并发运行，灰色 → 黑色，发现新对象标为灰色
3. STW：重新扫描（re-scan），处理并发期间的变更
4. 并发清除：回收白色对象

---

## 写屏障（Write Barrier）

并发标记期间，用户程序可能修改对象引用，写屏障保证不漏标。

**混合写屏障（Go 1.14+）：**
- 删除引用时，将被删除的对象标为灰色
- 新增引用时，将新对象标为灰色
- 消除了栈扫描的 STW，大幅降低延迟

---

## GC 触发条件

1. **堆内存增长**：当前堆大小达到上次 GC 后的 2 倍（由 `GOGC` 控制，默认 100）
2. **定时触发**：距上次 GC 超过 2 分钟
3. **手动触发**：`runtime.GC()`

```go
// GOGC=100 表示堆增长 100% 触发 GC
// GOGC=200 减少 GC 频率，增大内存占用
// GOGC=off 关闭 GC
```

## GC 调优

### sync.Pool 减少 GC 压力
```go
var pool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 1024)
    },
}

buf := pool.Get().([]byte)
// 使用 buf
pool.Put(buf) // 归还，GC 不会立即回收
```

### 减少内存分配
- 预分配 slice：`make([]T, 0, cap)`
- 复用对象：sync.Pool
- 避免频繁小对象分配（逃逸到堆）

### 逃逸分析
```bash
go build -gcflags="-m" ./...  # 查看逃逸情况
```

## 面试高频问题

**Q: Go GC 的 STW 时间有多长？**
- Go 1.14+ 通常 < 1ms，大多数情况 < 100μs
- 主要 STW 发生在初始标记阶段

**Q: 如何减少 GC 频率？**
- 增大 GOGC 值（减少触发频率，但内存占用增加）
- 使用 sync.Pool 复用对象
- 减少堆上分配（栈分配不触发 GC）

**Q: 什么是内存逃逸？**
- 本应在栈上分配的变量，因为生命周期超出函数范围，被分配到堆上
- 堆分配会增加 GC 压力
- 常见逃逸场景：返回局部变量指针、interface{} 参数、闭包捕获变量

**Q: 三色标记为什么需要写屏障？**
- 并发标记期间，用户程序可能将黑色对象引用白色对象（黑色不会再扫描）
- 写屏障拦截引用变更，将相关对象重新标灰，防止漏标导致存活对象被回收

**Q: GOGC 设置多少合适？**
- 延迟敏感：降低 GOGC（如 50），GC 更频繁但每次堆更小
- 吞吐量优先：增大 GOGC（如 200-400），减少 GC 次数
- 内存受限：降低 GOGC 或使用 `runtime/debug.SetMemoryLimit`（Go 1.19+）
