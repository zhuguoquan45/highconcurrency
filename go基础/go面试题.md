### **1. Go 并发模型是什么？goroutine 和线程有什么区别？**

* **考点**：Go 并发模型、Goroutine 与系统线程对比
* **答案**：

  * Goroutine 是 Go 的轻量级线程，由 Go runtime 调度，不依赖 OS 线程。
  * 栈初始约 2KB，可动态增长，线程栈一般为 MB 级别。
  * 调度器采用 **M:N 调度**（M 个 OS 线程调度 N 个 goroutine）。
* **原理解析**：

  * Goroutine 通过 runtime 的调度器调度，阻塞时会让出 CPU，不会阻塞整个线程。
* **面试思路提示**：

  * 面试官想考你对 Go 并发模型理解，是否知道 goroutine 轻量级特性和 M:N 调度。

### **2. channel 的底层原理是什么？如何实现线程安全？**

* **考点**：channel 原理、阻塞/非阻塞、线程安全
* **答案**：

  * channel 底层是带锁的队列结构（有缓冲或无缓冲）。
  * 发送/接收操作通过 mutex + wait queue 实现阻塞和唤醒。
  * 非阻塞通过 `select` 或 `default` case。
* **原理解析**：

  * goroutine 在 channel 上发送/接收时，如果阻塞，会挂起并加入等待队列，数据就绪时被唤醒。
* **面试思路提示**：

  * 测你是否理解 channel 的同步机制以及阻塞原理。

### **3. select 的底层实现及调度机制？**

* **考点**：select 原理、随机选择、阻塞机制
* **答案**：

  * select 会将每个 case 注册到 goroutine 的 wait queue 中。
  * runtime 随机选择就绪 case，避免饥饿。
  * 如果没有 case 就绪，goroutine 阻塞直到有 case 就绪。
* **原理解析**：

  * select 内部使用 `runtime.selectgo` 来管理 channel 队列，保证公平性。
* **面试思路提示**：

  * 面试官想看你是否能用 select 构建高并发任务处理。

### **4. Go 内存模型对并发安全的影响？**

* **考点**：Go Memory Model、happens-before、原子操作
* **答案**：

  * Go 保证在同步操作（如 channel、mutex、atomic）上的 **happens-before** 顺序。
  * 没有同步操作的读写可能会乱序，需要使用 `sync/atomic` 或 `mutex`。
* **原理解析**：

  * 内存屏障（Memory Barrier）保证可见性，避免 CPU 或编译器重排序影响并发。
* **面试思路提示**：

  * 考察你对共享内存并发安全的理解。

## **一、Channel & Select（题 11–18）**

**11. Go channel 分为哪几种？**

* **考点**：channel 类型与用途
* **答案**：

  * **无缓冲 channel**：发送阻塞直到接收方准备好
  * **有缓冲 channel**：发送方在缓冲未满时不会阻塞
  * **单向 channel**：只能发送或只能接收
* **原理解析**：无缓冲保证同步，有缓冲适合异步消息传递
* **面试提示**：测试 channel 类型理解和使用场景

**12. 如何实现非阻塞的 channel 发送或接收？**

* **考点**：非阻塞操作、select
* **答案**：

  ```go
  select {
  case ch <- value:
      // 成功发送
  default:
      // channel 满或接收方未就绪
  }
  ```
* **原理解析**：default 分支确保 goroutine 不阻塞
* **面试提示**：考察非阻塞控制和高并发处理能力

**13. select 的随机选择原理？**

* **考点**：select 调度机制
* **答案**：当多个 case 同时就绪时，Go runtime 随机选择一个执行，保证公平性
* **原理解析**：select 内部将 case 注册到 goroutine wait queue 并随机唤醒
* **面试提示**：测试底层调度理解

**14. 如何实现 channel 超时操作？**

* **考点**：context、time.After 与 select
* **答案**：

  ```go
  select {
  case val := <-ch:
      // 收到数据
  case <-time.After(1 * time.Second):
      // 超时
  }
  ```
* **原理解析**：time.After 返回一个 channel，超时后发送时间事件
* **面试提示**：常见高并发场景处理超时

**15. channel 的容量对性能有何影响？**

* **考点**：缓冲区大小与吞吐量
* **答案**：

  * 小容量 → 容易阻塞，提高同步保证
  * 大容量 → 提高吞吐量，但占用更多内存
* **原理解析**：容量越大，发送方阻塞概率越低，但堆积大量数据可能导致内存压力
* **面试提示**：考察对缓冲策略理解

**16. 如何实现 fan-out/fan-in 模式？**

* **考点**：goroutine 并发分发与汇总
* **答案**：

  * **Fan-out**：多个 worker goroutine 从同一个 channel 接收任务
  * **Fan-in**：多个 worker 输出结果到同一个 channel
* **原理解析**：利用 channel 作为共享队列，保证线程安全
* **面试提示**：高并发任务分发常用模式

**17. 如何优雅关闭 channel？**

* **考点**：channel 生命周期
* **答案**：

  * 发送方负责关闭 channel
  * 接收方使用 `for range ch` 循环读取直到关闭
* **原理解析**：关闭 channel 会触发接收方循环结束
* **面试提示**：避免 panic（向已关闭 channel 发送数据）

**18. 如何处理多个 channel 的合并（merge）？**

* **考点**：多 channel 合并
* **答案**：

  ```go
  func merge(cs ...<-chan int) <-chan int {
      out := make(chan int)
      var wg sync.WaitGroup
      for _, c := range cs {
          wg.Add(1)
          go func(c <-chan int) {
              defer wg.Done()
              for v := range c {
                  out <- v
              }
          }(c)
      }
      go func() {
          wg.Wait()
          close(out)
      }()
      return out
  }
  ```
* **原理解析**：fan-in + waitgroup + close 保证安全
* **面试提示**：面试官希望考察 channel 合并处理能力
