# 一、什么是缓存一致性

在互联网系统中，一般架构：

```
Client
  ↓
Cache (Redis)
  ↓
Database (MySQL/Postgres)
```

原因：

数据库慢，缓存快。

例如：

| 存储    | 延迟       |
| ----- | -------- |
| DB    | 5ms~20ms |
| Redis | 0.2ms    |

所以：

**绝大部分请求都应该走缓存。**

---

## 缓存一致性问题

当 **数据库和缓存数据不一致** 时就会发生。

例如：

```
DB     = 100
Cache  = 90
```

用户读到：

```
90
```

这是 **脏数据（Dirty Data）**。

---

# 二、缓存与数据库的三种关系

互联网系统有三种模式。

## 1 Cache Aside（旁路缓存） ⭐最常见

客户端自己控制缓存。

读流程：

```
read cache
   ↓ miss
read db
   ↓
write cache
```

写流程：

```
update db
delete cache
```

特点：

* 简单
* 性能高
* 最主流

几乎所有互联网系统：

```
淘宝
字节
美团
知乎
```

都是这个模式。

---

## 2 Read Through

缓存负责加载数据库。

```
client
  ↓
cache
  ↓
db
```

缓存 miss 时：

```
cache → db
```

优点：

逻辑简单

缺点：

缓存实现复杂。

互联网使用较少。

---

## 3 Write Through

写缓存同时写数据库。

```
client
  ↓
cache
  ↓
db
```

优点：

强一致。

缺点：

写性能差。

---

# 三、为什么推荐“更新DB + 删除缓存”

写流程：

```
update db
delete cache
```

很多初学者会问：

为什么不：

```
update db
update cache
```

原因有三个。

---

## 原因1：缓存可能不命中

假设：

```
update cache
```

但：

```
cache key 不存在
```

就会：

```
DB更新了
缓存没有
```

读请求：

```
读到旧缓存
```

---

## 原因2：多线程写冲突

假设：

```
线程A 更新缓存
线程B 更新缓存
```

顺序：

```
A update DB = 100
B update DB = 200
```

但缓存：

```
B update cache 200
A update cache 100
```

最终：

```
cache = 100
db = 200
```

数据错了。

---

## 原因3：缓存更新逻辑复杂

缓存可能来自：

```
多表 join
聚合
统计
```

更新成本很高。

删除缓存更简单。

---

# 四、更新DB + 删除缓存为什么仍然可能不一致

经典问题：

**读写并发**

时序：

```
线程B read cache miss
线程B read db (旧数据)

线程A update db
线程A delete cache

线程B write cache (旧数据)
```

结果：

```
cache = 旧数据
db = 新数据
```

---

# 五、延迟双删方案

解决办法：

```
update db
delete cache
sleep 500ms
delete cache
```

作用：

第二次删除清理脏数据。

时序：

```
线程B 写旧缓存
线程A 第二次删除
```

缓存恢复一致。

---

# 六、为什么很多公司不用延迟双删

原因：

1️⃣ 概率极低

需要同时满足：

```
缓存刚好过期
读线程正在查DB
写线程更新DB
```

概率很小。

---

2️⃣ sleep 不优雅

```
sleep 100ms ?
sleep 500ms ?
sleep 1s ?
```

很难确定。

---

3️⃣ 更好的方案

使用 **消息队列**

---

# 七、MQ保证缓存一致性（推荐方案）

架构：

```
update DB
delete cache
send MQ
consumer delete cache
```

流程：

```
业务服务
   ↓
更新数据库
   ↓
发送MQ
   ↓
缓存服务
   ↓
删除缓存
```

好处：

* 可重试
* 解耦
* 不阻塞线程

---

# 八、Binlog驱动缓存更新（大厂架构）

很多大厂使用 **CDC（Change Data Capture）**

架构：

```
MySQL
  ↓
Binlog
  ↓
Canal / Debezium
  ↓
MQ
  ↓
Cache Service
  ↓
Delete Cache
```

优点：

* 不侵入业务代码
* 自动同步
* 可扩展

# 九、缓存系统三大经典问题

## 1 缓存穿透

查询不存在数据：

```
user:999999
```

每次都访问DB。

解决：

```
布隆过滤器
缓存空值
```

---

## 2 缓存击穿

热点key过期：

```
商品详情
热门帖子
```

大量请求访问DB。

解决：

```
互斥锁
singleflight
```

---

## 3 缓存雪崩

大量key同时过期。

结果：

```
DB被打爆
```

解决：

```
随机TTL
分批过期
多级缓存
```

# 十二、面试标准回答（P7版本）

如果面试问：

**分布式缓存一致性怎么解决？**

标准回答：

> 互联网系统一般使用 Cache Aside 模式。
>
> 读流程是：
>
> ```
> 先读缓存
> miss再查数据库
> 再写入缓存
> ```
>
> 写流程一般是：
>
> ```
> 更新数据库
> 删除缓存
> ```
>
> 这样可以避免缓存和数据库的复杂更新逻辑。
>
> 在高并发情况下可能出现读写并发导致缓存脏数据，因此可以使用：
>
> * 延迟双删
> * MQ再次删除缓存
> * binlog订阅删除缓存
>
> 大型系统一般通过 **MQ 或 binlog CDC** 保证最终一致性。

# 十三、一句话总结

缓存系统本质是：

```
最终一致系统
```

而不是：

```
强一致系统
```

允许 **短暂不一致**，但要保证 **最终一致**。
