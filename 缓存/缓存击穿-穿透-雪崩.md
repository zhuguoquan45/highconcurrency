缓存这三个问题经常被一起问：**缓存穿透、缓存击穿、缓存雪崩**。
很多初学者一开始会混淆，其实可以用一句话先记住核心区别：

> **穿透：查不到的数据**
> **击穿：某个热点数据突然失效**
> **雪崩：大量缓存同时失效**

下面我用**非常直观的方式 + 简单架构图思维 + 解决方案**给你梳理一遍。🚀

---

# 一、缓存穿透（Cache Penetration）

## 1 什么是缓存穿透

请求查询一个 **数据库中根本不存在的数据**。

流程：

```
请求 -> Redis缓存 -> 没有
        -> 数据库 -> 没有
```

由于数据库也没有，所以：

* Redis不会缓存
* 每次请求都会打到数据库

如果攻击者不断请求不存在的数据，就会造成 **数据库压力巨大**。

---

## 2 举例

查询用户：

```
GET /user?id=100000000
```

数据库只有：

```
1-10000
```

攻击者不断请求：

```
user:999999999
user:888888888
user:777777777
```

每次都会：

```
请求 -> Redis MISS -> DB -> MISS
```

数据库被打爆。

---

## 3 解决方案

### 方案1：缓存空值（最简单）

如果数据库查询为空：

```
SET user:999999 null EX 60
```

缓存一个空值。

以后再查：

```
Redis -> null -> 直接返回
```

优点

* 简单
* 80%场景够用

缺点

* 会产生一些无效缓存

---

### 方案2：布隆过滤器（生产常用）

在 Redis 前增加：

```
BloomFilter
```

流程：

```
请求
  ↓
BloomFilter判断
  ↓
不存在 → 直接返回
存在 → 查Redis
```

示意：

```
          ┌──────────┐
请求 ---> │ BloomFilter │
          └─────┬────┘
                │
          不存在│存在
                │
             返回错误
                │
             Redis
                │
             Database
```

优点：

* 不会访问数据库
* 防止攻击

缺点：

* 有 **极小概率误判**

---

# 二、缓存击穿（Cache Breakdown）

## 1 什么是缓存击穿

**某一个热点Key突然过期**

在过期瞬间：

```
大量请求同时访问数据库
```

流程：

```
大量请求
     ↓
Redis MISS
     ↓
全部访问 DB
```

数据库瞬间被压垮。

---

## 2 举例

热点商品：

```
product:1001
```

缓存：

```
TTL = 30min
```

30分钟到期那一刻：

```
10万请求同时进来
```

发生：

```
10万次 DB 查询
```

数据库扛不住。

---

## 3 解决方案

### 方案1：互斥锁（最常见）

只有一个线程查询数据库。

流程：

```
请求
  ↓
Redis MISS
  ↓
尝试加锁
  ↓
成功 → 查询DB → 写缓存
失败 → 等待缓存
```

伪代码：

```go
val := redis.Get(key)

if val == nil {

    if tryLock(key) {

        val = db.Query()

        redis.Set(key, val)

        unlock(key)

    } else {

        sleep(50ms)
        return redis.Get(key)

    }

}
```

效果：

```
10000请求
   ↓
只有1个查DB
```

---

### 方案2：逻辑过期（高并发推荐）

缓存永不过期，但加一个 **逻辑时间**。

数据结构：

```
{
   data: 商品数据
   expire: 2026-03-11 10:00
}
```

流程：

```
请求
  ↓
Redis读取
  ↓
是否过期？
   ↓
否 → 直接返回
是 → 后台线程更新
```

优点：

* 用户永远能拿到数据
* 不会打爆数据库

---

# 三、缓存雪崩（Cache Avalanche）

## 1 什么是缓存雪崩

**大量缓存同时失效**

例如：

```
10万个key
TTL = 1小时
```

1小时后：

```
全部过期
```

瞬间：

```
所有请求打到数据库
```

数据库直接崩。

---

## 2 举例

电商首页：

```
banner
商品列表
推荐商品
分类
库存
```

全部缓存：

```
TTL = 3600
```

一小时后：

```
所有缓存一起失效
```

数据库瞬间压力：

```
100倍
```

---

## 3 解决方案

### 方案1：随机过期时间（最常见）

不要统一TTL：

```
3600 + rand(0-300)
```

示例：

```
key1 3605
key2 3710
key3 3620
key4 3890
```

过期时间被打散。

---

### 方案2：多级缓存

架构：

```
Browser
   ↓
CDN
   ↓
Nginx缓存
   ↓
Redis
   ↓
Database
```

就算 Redis 挂：

```
CDN / Nginx还能扛一层
```

---

### 方案3：缓存预热

系统启动时提前加载：

```
Redis.set(...)
```

避免第一次请求直接打 DB。

---

# 四、三者区别总结

| 问题   | 原因        | 特点        |
| ---- | --------- | --------- |
| 缓存穿透 | 查询不存在的数据  | 每次请求都到DB  |
| 缓存击穿 | 热点Key过期   | 瞬间大量请求打DB |
| 缓存雪崩 | 大量Key同时过期 | 数据库瞬间压力巨大 |

一句话记忆：

```
穿透 → 查不到
击穿 → 单点热点
雪崩 → 大面积失效
```

---

# 五、真实生产架构（互联网常见）

一般会组合使用：

```
请求
  ↓
布隆过滤器
  ↓
Redis
  ↓
互斥锁
  ↓
数据库
```

并且：

```
TTL = base + random
```

再加：

```
多级缓存 + 限流
```

完整防御体系：

```
穿透 → BloomFilter
击穿 → Mutex Lock
雪崩 → TTL Random
```

# 六、面试高频问题（非常常见）

面试官常问：

### 问1

缓存穿透和击穿区别？

答：

```
穿透：查询不存在的数据
击穿：热点key失效
```

---

### 问2

为什么布隆过滤器会误判？

答：

```
哈希冲突导致
```

---

### 问3

为什么要随机TTL？

答：

```
避免缓存同时过期导致雪崩
```
