## 一、PostgreSQL 支持哪些事务隔离级别？

PostgreSQL **支持 SQL 标准定义的 4 种隔离级别**（名字都一样）：

| 隔离级别             | PG 是否支持                | 实现方式                                 |
| ---------------- | ---------------------- | ------------------------------------ |
| READ UNCOMMITTED | ✅（但等同于 READ COMMITTED） | MVCC（多版本并发控制： 读老写新）                                 |
| READ COMMITTED   | ✅（默认）                  | MVCC                                 |
| REPEATABLE READ  | ✅                      | Snapshot Isolation                   |
| SERIALIZABLE     | ✅                      | SSI（Serializable Snapshot Isolation） |

⚠️ **PG 不存在真正的“脏读”**

---

## 二、四个隔离级别分别“看到什么”？

### 1️⃣ READ UNCOMMITTED（读未提交）

```sql
BEGIN ISOLATION LEVEL READ UNCOMMITTED;
```

📌 **在 PostgreSQL 中：等同于 READ COMMITTED**

* 永远读不到未提交数据
* PG 直接忽略这个级别

👉 原因：**MVCC 天生不允许脏读**

---

### 2️⃣ READ COMMITTED（默认，最常用）

```sql
BEGIN;
SELECT ...
```

#### 行为特征

* **每条 SQL 都用一个新快照**
* 能看到**其他事务已提交的新数据**
* 同一个事务里：

  ```text
  第一次 SELECT ≠ 第二次 SELECT
  ```

#### 会出现的问题

* ❌ 不可重复读
* ❌ 幻读

#### 示例

```text
T1: SELECT count(*) FROM orders; -- 10
T2: INSERT INTO orders VALUES (...); COMMIT;
T1: SELECT count(*) FROM orders; -- 11
```

---

### 3️⃣ REPEATABLE READ（可重复读）

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
```

#### 行为特征（非常重要）

* **事务开始时创建一次快照**
* 整个事务都用这个快照
* 多次 SELECT 结果一致

#### 能解决什么？

* ✅ 不可重复读
* ✅ 幻读（在 PG 里）

⚠️ 和 MySQL InnoDB 不同
PG 的 REPEATABLE READ **真的不会幻读**

---

### 4️⃣ SERIALIZABLE（可串行化）

```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;
```

#### PostgreSQL 的实现方式

> **SSI（Serializable Snapshot Isolation）**

* 不靠大锁
* 不阻塞
* **靠冲突检测 + 回滚事务**

#### 行为特征

* 效果等价于：

  ```text
  所有事务一个接一个执行
  ```
* 但实际上仍是并发执行

#### 特殊点 ⚠️

* 可能在 **COMMIT 时失败**

```text
ERROR: could not serialize access due to read/write dependencies
```

👉 需要应用层 **重试事务**

## 三、经典并发问题对比表（PG 视角）

| 问题     | RC   | RR  | Serializable |
| ------ | ---- | --- | ------------ |
| 脏读     | ❌    | ❌   | ❌            |
| 不可重复读  | ❌    | ✅   | ✅            |
| 幻读     | ❌    | ✅   | ✅            |
| 事务回滚概率 | 低    | 低   | 高            |
| 并发性能   | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐           |

## 四、PG vs MySQL(InnoDB) 的一个大坑区别 ⚠️

| 点               | PostgreSQL | MySQL InnoDB  |
| --------------- | ---------- | ------------- |
| RR 是否防幻读        | ✅（靠 MVCC）  | ❌（靠 gap lock） |
| Gap Lock        | ❌          | ✅             |
| Serializable 实现 | SSI        | 强锁            |
| RC 默认           | ❌          | ❌             |
| 默认隔离级别          | RC         | RR            |

👉 **PG 没有 gap lock，这是设计差异**

## 五、什么时候该用哪个？

### 🔹 READ COMMITTED（90% 场景）

* Web API
* OLTP
* 高并发写

✅ 性能最好
❌ 要自己处理并发一致性（版本号、乐观锁）

---

### 🔹 REPEATABLE READ

* 报表
* 复杂查询
* 一致性要求高但能接受稍低并发

---

### 🔹 SERIALIZABLE

* 金融
* 账务
* 强一致业务

⚠️ **必须有重试机制**

## 六、怎么查看 & 设置事务隔离级别？

### 查看当前级别

```sql
SHOW transaction_isolation;
```

### 设置

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
```

或

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

## **一、事务与 MVCC（题 1–8）**

**1. PostgreSQL 的事务隔离级别有哪些？区别？**

* **考点**：事务隔离
* **答案**：

  * **Read Uncommitted**：允许脏读（PG 会自动降级到 Read Committed）
  * **Read Committed**：默认，读到提交的最新数据
  * **Repeatable Read**：同一事务内多次读取结果一致
  * **Serializable**：完全串行化，避免并发冲突
* **原理解析**：MVCC + Snapshot 实现多版本控制
* **面试提示**：事务隔离理解能力

**2. PostgreSQL 的 MVCC 是如何实现的？**

* **考点**：多版本并发控制
* **答案**：

  * 每行数据有 xmin/xmax 标记事务生效范围
  * 查询时通过 Snapshot 判断数据是否可见
* **原理解析**：读不阻塞写，写不阻塞读
* **面试提示**：高并发核心机制

**3. MVCC 如何解决脏读、不可重复读、幻读？**

* **考点**：事务可见性
* **答案**：

  * 脏读：Read Committed 防止
  * 不可重复读：Repeatable Read
  * 幻读：Serializable
* **原理解析**：事务 snapshot 与行版本控制
* **面试提示**：隔离级别选择场景

**4. VACUUM 的作用是什么？**

* **考点**：存储清理
* **答案**：

  * 清理已过期行版本，回收空间
  * 防止表膨胀
* **原理解析**：MVCC 更新删除产生 dead tuple
* **面试提示**：维护高并发性能

**5. autovacuum 如何工作？**

* **考点**：自动维护
* **答案**：

  * 后台自动扫描表，清理 dead tuple
  * 配置 threshold 和 scale factor 调整触发条件
* **原理解析**：保证表空间和索引可用
* **面试提示**：运维优化能力

**6. PostgreSQL 如何保证事务原子性？**

* **考点**：ACID
* **答案**：

  * WAL（Write-Ahead Log）保证 crash 恢复
  * BEGIN…COMMIT/ROLLBACK 事务控制
* **原理解析**：日志先写入磁盘，再修改数据
* **面试提示**：事务安全理解

**7. 如何解决 PostgreSQL 高并发写入冲突？**

* **考点**：并发写优化
* **答案**：

  * 使用行级锁（SELECT … FOR UPDATE）
  * 乐观锁（版本号或 timestamp）
  * 避免大事务长时间占用锁
* **原理解析**：行级锁 + MVCC 减少冲突
* **面试提示**：写冲突处理

**8. repeatable read 与 serializable 的性能区别？**

* **考点**：隔离级别性能
* **答案**：

  * Repeatable Read：无需阻塞，只读 snapshot
  * Serializable：可能回滚事务，性能低
* **原理解析**：Serializable 使用 Serializable Snapshot Isolation (SSI)
* **面试提示**：隔离 vs 性能权衡

## **二、锁机制（题 9–14）**

**9. PostgreSQL 支持哪些锁类型？**

* **考点**：锁机制
* **答案**：

  * 行级锁（FOR UPDATE / FOR SHARE）
  * 表级锁（ACCESS EXCLUSIVE / SHARE）
  * Advisory Lock（应用级自定义锁）
* **原理解析**：行级锁粒度小，表级锁粒度大
* **面试提示**：高并发锁理解

**10. 行级锁和表级锁的区别？**

* **考点**：锁粒度
* **答案**：

  * 行锁：锁单条记录，冲突少
  * 表锁：锁整个表，冲突高
* **原理解析**：MVCC + 锁粒度决定并发性能
* **面试提示**：设计并发控制策略

**11. 什么是 Advisory Lock？应用场景？**

* **考点**：应用级锁
* **答案**：

  * 用户自定义锁，不阻塞事务提交
  * 用于分布式任务调度、批量处理互斥
* **原理解析**：存储锁状态在 shared memory
* **面试提示**：高并发任务协调

**12. 如何检测锁冲突或死锁？**

* **考点**：死锁排查
* **答案**：

  * pg_locks 查看锁状态
  * pg_stat_activity 查看阻塞情况
  * Deadlock 日志分析
* **原理解析**：锁等待图分析
* **面试提示**：生产运维经验

**13. PostgreSQL 的锁是悲观锁还是乐观锁？**

* **考点**：并发控制策略
* **答案**：

  * 默认使用乐观锁（MVCC 读不阻塞写）
  * 悲观锁通过 SELECT FOR UPDATE 实现
* **原理解析**：读写冲突减少，保证性能
* **面试提示**：锁策略选择

**14. 高并发下如何避免行锁热点？**

* **考点**：热点数据优化
* **答案**：

  * 分散 key（hash sharding）
  * 使用批量更新而非频繁更新同一行
* **原理解析**：减少单行锁等待
* **面试提示**：高并发设计
