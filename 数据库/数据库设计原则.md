# 高并发 Go 系统数据库设计综合指南

---

## 1️⃣ 数据库设计目标

在高并发系统中，数据库设计要兼顾：

1. **数据安全与一致性**
2. **高性能读写**
3. **可扩展性**（支持分库分表、水平扩展）
4. **可维护性**（后续功能扩展方便）

---

## 2️⃣ 规范化与反规范化

| 类型   | 特点             | 适用场景       |
| ---- | -------------- | ---------- |
| 规范化  | 减少冗余，提高一致性     | 数据量小或写多的系统 |
| 反规范化 | 提高查询效率，减少 join | 高并发读多的系统   |

> 面试答题示例：
> “高并发系统中常做折中处理，核心表保持规范化，热点查询表可以适度反规范化。”

---

## 3️⃣ 拆分策略

### 垂直拆分（Vertical Split）

* 按功能拆表，减少单表列数
* 优点：查询快、表小
* 示例：

```
用户表: [ID, Name, Email]
订单表: [ID, UserID, ProductID, Amount]
商品表: [ID, Name, Price]
```

### 水平拆分（Sharding）

* 按用户ID或时间拆表/数据库
* 优点：减少单表压力，支持更多并发
* 示例：

```
订单表_0: 用户ID % 3 = 0
订单表_1: 用户ID % 3 = 1
订单表_2: 用户ID % 3 = 2
```

---

## 4️⃣ 主键设计

| 类型   | 优点       | 注意点         |
| ---- | -------- | ----------- |
| 自增ID | 简单，顺序    | 高并发写入可能成为热点 |
| UUID | 全局唯一     | 占空间、索引效率略低  |
| 雪花ID | 高并发分布式唯一 | 需要算法实现      |

💡 高并发 Go 系统常用 **雪花ID + 分库分表**。

---

## 5️⃣ 索引设计

* **作用**：快速定位数据（类似书的目录）
* **策略**：

  1. 常用查询字段加索引
  2. 组合索引应对多条件查询
  3. 高并发写多场景要权衡索引数量

---

## 6️⃣ 缓存与分布式设计

* **Redis / Memcached** 用于热点数据缓存
* **缓存策略**：

  * Cache Aside：先查缓存，没数据再查数据库
  * Write Through：写数据库同时更新缓存
* **读写分离**：

  * 主库写入
  * 从库读取，提高并发

---

## 7️⃣ 乐观锁 & 悲观锁

| 类型  | 特点                | 适用场景     |
| --- | ----------------- | -------- |
| 悲观锁 | 操作前锁住数据，防止其他操作修改  | 写冲突频繁    |
| 乐观锁 | 不锁数据，更新时验证版本号或时间戳 | 高并发写少，读多 |

💡 秒杀系统常用 **乐观锁 + Redis 预扣库存**，防止超卖。

---

## 8️⃣ 数据归档与分区

* 历史数据或冷数据存到归档库或分区表
* 提高热点表性能
* 示例：

```
订单表_2026_01 订单表_2026_02 ...
```

---

## 9️⃣ 实战示例：秒杀系统数据库设计

1. **Redis缓存库存**
2. **创建订单**：写数据库
3. **更新库存**：数据库 + Redis
4. **防止超卖**：乐观锁 / 版本号检查

💡 图示流程：

```
用户下单 → Redis预扣库存 → 数据库写订单 → Redis更新库存 → 用户成功/失败
```
