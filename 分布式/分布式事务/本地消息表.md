**本地消息表（Local Message Table）** 是微服务里实现 **最终一致性（Eventual Consistency）** 的一种非常经典的分布式事务方案。很多互联网系统（电商、支付、订单系统）都会用它。核心思想其实很简单：

> **业务数据更新 + 发送消息 = 同一个本地事务**

这样就避免了“业务成功但消息没发出去”的问题。

我给你系统讲一遍，从 **为什么需要 → 原理 → 流程 → 表结构 → Go 示例 → 优缺点**。

---

# 一、为什么需要本地消息表

假设一个典型微服务场景：

**订单服务创建订单 → 通知库存服务扣库存**

简单实现可能是：

```
1 创建订单
2 调用MQ发送消息
3 库存服务消费消息
```

问题是：

| 情况            | 结果    |
| ------------- | ----- |
| 订单写成功，MQ发送失败  | 库存不会扣 |
| MQ发送成功，订单事务回滚 | 库存却扣了 |
| 网络异常          | 数据不一致 |

**核心问题：**

```
数据库事务 和 MQ 事务不是一个事务
```

所以需要一种机制：

```
保证：
订单成功  -> 消息一定能发
订单失败  -> 消息一定不会发
```

这就是 **本地消息表方案**。

---

# 二、本地消息表核心思想

核心原则：

```
业务操作 + 写消息表
必须在同一个数据库事务
```

流程：

```
业务表
消息表
```

一起提交。

---

# 三、本地消息表架构

```
             ┌───────────────┐
             │ 订单服务       │
             └──────┬────────┘
                    │
                    │本地事务
                    ▼
        ┌────────────────────────┐
        │      数据库            │
        │                        │
        │ order表                │
        │ message表              │
        └──────────┬─────────────┘
                   │
                   │ 定时任务扫描
                   ▼
             ┌───────────┐
             │   MQ       │
             └─────┬──────┘
                   │
                   ▼
             库存服务消费
```

关键点：

```
消息先落库，再异步发送
```

# 四、完整执行流程

### Step1 订单创建

开启事务：

```
BEGIN
```

执行两件事：

```
1 写订单表
2 写消息表
```

```
INSERT order
INSERT message
```

提交事务：

```
COMMIT
```

此时：

```
订单一定存在
消息一定存在
```

但消息 **还没有发送**。

---

### Step2 定时任务扫描消息表

后台任务扫描：

```
status = pending
```

发送消息到 MQ：

```
Send MQ
```

发送成功：

```
update message set status = sent
```

失败：

```
等待重试
```

---

### Step3 消费端处理

库存服务：

```
消费MQ
扣库存
```

最好做：

```
幂等控制
```

例如：

```
订单ID唯一
```

避免重复消费。

---

# 五、消息表设计

典型结构：

```sql
CREATE TABLE message (
    id BIGSERIAL PRIMARY KEY,

    event_type VARCHAR(50),

    payload JSONB,

    status VARCHAR(20),

    retry_count INT DEFAULT 0,

    next_retry TIMESTAMP,

    created_at TIMESTAMP,

    updated_at TIMESTAMP
);
```

字段说明：

| 字段          | 作用             |
| ----------- | -------------- |
| event_type  | 事件类型           |
| payload     | 消息内容           |
| status      | pending / sent |
| retry_count | 重试次数           |
| next_retry  | 下次重试时间         |

# 六、Go + PostgreSQL 示例

### 1 创建订单 + 写消息

```go
func CreateOrder(db *sql.DB) error {

    tx, err := db.Begin()
    if err != nil {
        return err
    }

    // 写订单
    _, err = tx.Exec(`
        INSERT INTO orders(id, user_id, amount)
        VALUES($1,$2,$3)
    `, 1001, 1, 200)

    if err != nil {
        tx.Rollback()
        return err
    }

    // 写消息表
    msg := `{"order_id":1001}`

    _, err = tx.Exec(`
        INSERT INTO message(event_type,payload,status)
        VALUES($1,$2,'pending')
    `, "order_created", msg)

    if err != nil {
        tx.Rollback()
        return err
    }

    return tx.Commit()
}
```

关键点：

```
订单 + 消息
一个事务
```

---

### 2 定时任务发送消息

```go
func PublishMessage(db *sql.DB) {

    rows, _ := db.Query(`
        SELECT id, payload
        FROM message
        WHERE status='pending'
        LIMIT 100
    `)

    for rows.Next() {

        var id int
        var payload string

        rows.Scan(&id, &payload)

        err := SendMQ(payload)

        if err == nil {
            db.Exec(`
                UPDATE message
                SET status='sent'
                WHERE id=$1
            `, id)
        }
    }
}
```

# 七、消息可靠性机制

生产系统会加这些：

### 1 消息重试

```
retry_count + next_retry
```

失败：

```
retry_count++
next_retry = now + 1min
```

---

### 2 死信队列

如果：

```
retry_count > 10
```

进入：

```
dead letter queue
```

人工处理。

---

### 3 幂等消费

消费端必须支持：

```
重复消费
```

常见做法：

```
唯一业务ID
```

例如：

```
order_id 唯一
```

# 八、本地消息表 vs 2PC

| 方案    | 特点       |
| ----- | -------- |
| 2PC   | 强一致，但性能差 |
| 本地消息表 | 最终一致，高性能 |
| Saga  | 业务补偿     |
| TCC   | 强业务侵入    |

互联网最常见的是：

```
本地消息表 + MQ
```

# 十、本地消息表的优缺点

## 优点

✔ 不需要分布式事务
✔ 高性能
✔ 实现简单
✔ MQ解耦

---

## 缺点

❌ 最终一致性
❌ 消息表需要维护
❌ 定时任务增加复杂度

# 十一、进阶架构（大厂版本）

### Outbox Pattern

结构：

```

outbox表
```

然后：

```
Debezium
Canal

```

监听 binlog：

```
binlog → MQ
```

架构：

```
DB binlog
   ↓
CDC
   ↓
Kafka
```

优点：

```
无需定时任务
实时发送
```

# 十二、面试总结（非常经典）

一句话总结：

```
本地消息表通过把业务数据和消息记录放在同一个本地事务中，
再由后台任务异步发送消息，从而保证消息一定发送，实现分布式系统最终一致性。
```

核心步骤：

```
1 业务操作
2 写消息表
3 提交事务
4 定时任务发送MQ
5 消费者幂等处理
```
