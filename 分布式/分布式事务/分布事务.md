# 微服务分布式事务学习文档（入门到实践）

## 1 什么是分布式事务

在单体系统中，一个事务通常在 **一个数据库连接里完成**：

```sql
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id=1;
UPDATE account SET balance = balance + 100 WHERE id=2;
COMMIT;
```

如果中途失败：

```
ROLLBACK;
```

数据库保证 **ACID**：

| 特性          | 说明  |
| ----------- | --- |
| Atomicity   | 原子性 |
| Consistency | 一致性 |
| Isolation   | 隔离性 |
| Durability  | 持久性 |

---

但在 **微服务架构** 中：

```
订单服务
支付服务
库存服务
账户服务
```

每个服务：

```
独立数据库
独立事务
```

例如下单流程：

```
订单服务  -> 创建订单
库存服务  -> 扣库存
支付服务  -> 扣钱
积分服务  -> 加积分
```

问题：

```
订单成功
库存扣了
支付失败
```

系统数据就 **不一致**。

这就是：

**分布式事务问题**

---

# 2 为什么微服务一定会遇到分布式事务

微服务有一个原则：

**Database per service**

每个服务：

```
独立数据库
独立事务
```

所以：

```
一个业务 = 多个数据库事务
```

例如：

```
用户下单

订单服务  -> order_db
库存服务  -> stock_db
支付服务  -> pay_db
```

一个业务跨：

```
多个服务
多个数据库
多个事务
```

# 4 两阶段提交（2PC）

2PC 是最经典的分布式事务方案。

### 角色

```
协调者（Coordinator）
参与者（Participant）
```

流程：

```
      Coordinator
           |
    -----------------
    |               |
 Participant1   Participant2
```

---

## 第一阶段：Prepare

协调者询问各个节点：

```
是否可以提交？
```

流程：

```
Coordinator -> P1 prepare
Coordinator -> P2 prepare
```

参与者：

```
执行事务
写undo log
不提交
返回 YES / NO
```

---

## 第二阶段：Commit

如果全部返回 YES：

```
Coordinator -> commit
```

否则：

```
Coordinator -> rollback
```

---

### 2PC流程图

```
        Prepare阶段

  Coordinator
      |
      |---- prepare ----> P1
      |---- prepare ----> P2
      |
      |<---- yes --------|
      |<---- yes --------|

        Commit阶段

      |---- commit -----> P1
      |---- commit -----> P2
```

---

## 2PC的问题

主要问题：

### 1 阻塞问题

如果 Coordinator 挂了：

```
参与者不知道 commit 还是 rollback
```

数据库：

```
锁住
```

系统阻塞。

---

### 2 单点问题

Coordinator 是：

```
单点
```

---

### 3 性能差

因为：

```
锁资源
等待网络
```

高并发系统几乎不用。

---

# 5 三阶段提交（3PC）

3PC 在 2PC 基础上增加：

```
CanCommit
PreCommit
DoCommit
```

流程：

```
CanCommit -> 询问是否可以提交
PreCommit -> 预提交
DoCommit  -> 正式提交
```

优点：

```
减少阻塞
```

但：

```
实现复杂
很少使用
```

---

# 6 TCC事务（互联网最常用）

TCC：

```
Try
Confirm
Cancel
```

思想：

```
业务补偿
```

---

## TCC流程

假设：

```
下单
扣库存
扣余额
```

### Try

预留资源：

```
冻结库存
冻结余额
```

---

### Confirm

真正执行：

```
扣库存
扣钱
```

---

### Cancel

失败回滚：

```
解冻库存
解冻余额
```

---

## TCC流程图

```
          Try阶段

订单服务 -> 库存服务 Try
订单服务 -> 账户服务 Try

          Confirm阶段

订单服务 -> 库存 Confirm
订单服务 -> 账户 Confirm
```

如果失败：

```
Cancel
```

---

## TCC代码示例（Go）

Try阶段：

```go
func TryDeductStock(productId int, count int) error {

    // 冻结库存
    _, err := db.Exec(`
        UPDATE stock 
        SET freeze = freeze + ?, available = available - ?
        WHERE product_id=? AND available >= ?
    `, count, count, productId, count)

    return err
}
```

Confirm：

```go
func ConfirmDeductStock(productId int, count int) error {

    _, err := db.Exec(`
        UPDATE stock
        SET freeze = freeze - ?, used = used + ?
        WHERE product_id = ?
    `, count, count, productId)

    return err
}
```

Cancel：

```go
func CancelDeductStock(productId int, count int) error {

    _, err := db.Exec(`
        UPDATE stock
        SET freeze = freeze - ?, available = available + ?
        WHERE product_id = ?
    `, count, count, productId)

    return err
}
```

---

## TCC优点

优点：

```
高性能
无锁
互联网广泛使用
```

缺点：

```
代码侵入大
业务复杂
```

# 7 Saga事务（微服务常用）

Saga 是：

```
长事务解决方案
```

思想：

```
一系列本地事务 + 补偿事务
```

流程：

```
T1 -> T2 -> T3
```

如果：

```
T3失败
```

执行补偿：

```
C2 -> C1
```

---

### Saga流程

```
订单创建
   |
扣库存
   |
扣余额
```

失败：

```
退款
恢复库存
取消订单
```

---

### Saga流程图

```
Order -> Stock -> Payment
  |        |         |
  v        v         v
 C1       C2        C3
```

---

## Saga优点

优点：

```
适合长事务
高扩展
```

缺点：

```
数据短暂不一致
```

# 8 本地消息表（阿里经典方案）

核心思想：

```
事务 + 消息
```

保证：

```
消息不丢
```

流程：

```
1 本地事务
2 写消息表
3 提交事务
4 MQ发送
```

---

### 表结构

```sql
CREATE TABLE message (
    id bigint primary key,
    topic varchar(64),
    body text,
    status int
);
```

---

### 事务流程

```
BEGIN

订单创建
写消息表

COMMIT
```

后台任务：

```
扫描消息表
发送MQ
更新状态
```

---

# 9 MQ事务（RocketMQ）

RocketMQ支持：

```
事务消息
```

流程：

```
发送 half message
执行本地事务
提交 / 回滚
```

流程图：

```
Producer -> MQ (half message)

执行本地事务

成功 -> commit
失败 -> rollback
```
