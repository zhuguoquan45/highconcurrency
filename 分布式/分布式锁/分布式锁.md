# Go 语言分布式锁综合文档

---

## 一、分布式锁概述

### 1. 什么是分布式锁

* 分布式锁是跨多个进程或服务实例的互斥机制
* 用于保证在 **分布式环境下同一资源同时只有一个客户端操作**
* 解决场景：

  * 库存扣减防止超卖
  * 分布式定时任务只执行一次
  * 防止重复提交、并发操作冲突

### 2. 与本地锁对比

| 类型   | 作用范围    | 实现示例                     |
| ---- | ------- | ------------------------ |
| 本地锁  | 单进程     | `sync.Mutex`             |
| 分布式锁 | 多实例、多节点 | Redis / ZooKeeper / etcd |

---

## 二、分布式锁核心特性

1. **互斥性**：同一时间只有一个客户端持有锁
2. **防死锁**：锁超时自动释放
3. **可重入性**（可选）：同一个线程可多次加锁
4. **高可用**：锁服务不能单点失效

---

## 三、Redis 分布式锁

### 1. 获取锁

```go
SET key value NX PX ttl
```

* NX：不存在时设置
* PX：设置过期时间

### 2. 释放锁

错误方式：

```go
DEL key
```

问题：可能删除别人锁

正确方式：使用 Lua 脚本保证原子性

```lua
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end
```

---

### 3. 可重入锁

* 同一个客户端可以多次加锁
* 实现方法：

  * value 中保存客户端标识和计数器
  * Lua 脚本判断是否持有锁，更新计数器
  * 解锁时计数器归零才释放

---

### 4. 阻塞锁 / 等待锁

* 阻塞锁：获取不到锁就等待，直到超时或成功
* 非阻塞锁（TryLock）：获取不到锁直接返回失败
* Redis 实现方法：

  * 轮询 + Sleep
  * 或者结合 Pub/Sub 通知（复杂）

---

### 5. 锁续期（WatchDog）

* 防止业务执行时间超过锁 TTL 导致锁过期
* 核心：

  1. 后台协程定期检查锁是否属于自己
  2. 如果是 → 延长 TTL
* Lua 脚本原子操作：

```lua
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("pexpire",KEYS[1], ARGV[2])
else
    return 0
end
```

* 避免无限续期：

  * 绑定业务线程生命周期
  * 校验 value
  * 设置最大续期时间
  * WatchDog 停止时 TTL 到期锁释放

---

### 6. Go 实现示例

```go
import (
    "context"
    "fmt"
    "time"
    "github.com/google/uuid"
    "github.com/redis/go-redis/v9"
)

func AcquireLock(ctx context.Context, rdb *redis.Client, key string, value string, ttl time.Duration) (bool, error) {
    return rdb.SetNX(ctx, key, value, ttl).Result()
}

var unlockScript = redis.NewScript(`
if redis.call("get",KEYS[1]) == ARGV[1]
then
    return redis.call("del",KEYS[1])
else
    return 0
end
`)

func ReleaseLock(ctx context.Context, rdb *redis.Client, key, value string) error {
    _, err := unlockScript.Run(ctx, rdb, []string{key}, value).Result()
    return err
}
```

**WatchDog 续期示例：**

```go
func StartWatchDog(ctx context.Context, rdb *redis.Client, key, value string, ttl time.Duration) {
    ticker := time.NewTicker(ttl / 3)
    go func() {
        for {
            select {
            case <-ticker.C:
                _ = redis.NewScript(`
                if redis.call("get",KEYS[1]) == ARGV[1]
                then
                    return redis.call("pexpire", KEYS[1], ARGV[2])
                else
                    return 0
                end`).Run(ctx, rdb, []string{key}, value, ttl.Milliseconds()).Err()
            case <-ctx.Done():
                ticker.Stop()
                return
            }
        }
    }()
}
```

## 五、ZooKeeper / etcd 分布式锁

### 1. ZooKeeper

* 使用**临时顺序节点**实现锁
* 最小节点获得锁，其余监听前驱节点
* 优点：强一致性
* 缺点：性能不如 Redis

### 2. etcd

* 使用 `lease + key` 实现
* lease 到期自动释放锁
* Go 生态常用 `clientv3/concurrency` 实现分布式锁

## 六、生产最佳实践

1. **锁必须有过期时间**，防止死锁
2. **使用唯一 value** 防止误删别人锁
3. **Lua 脚本释放锁**，保证原子性
4. **业务执行时间不能超过锁 TTL**，或使用 WatchDog
5. **高并发场景**：

   * Redis 单节点 → bsm/redis-lock
   * Redis 集群或多节点 → redsync/Redlock
   * 高一致性需求 → ZooKeeper 或 etcd
