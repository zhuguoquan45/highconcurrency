# 限流与熔断

## 限流（Rate Limiting）

### 为什么需要限流
- 保护下游服务不被流量压垮
- 防止恶意请求、爬虫
- 保证核心服务的资源分配

---

### 令牌桶算法（Token Bucket）

**原理：** 以固定速率往桶里放令牌，请求消耗令牌，桶满则丢弃新令牌。允许突发流量。

```
令牌生成速率: r tokens/s
桶容量: b tokens
请求到来时取一个令牌，无令牌则拒绝
```

```go
// 使用 golang.org/x/time/rate
import "golang.org/x/time/rate"

limiter := rate.NewLimiter(rate.Limit(100), 200) // 100 req/s，桶容量 200

func handler(w http.ResponseWriter, r *http.Request) {
    if !limiter.Allow() {
        http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
        return
    }
    // 处理请求
}

// 等待令牌（阻塞）
ctx, cancel := context.WithTimeout(r.Context(), time.Second)
defer cancel()
if err := limiter.Wait(ctx); err != nil {
    // 超时或取消
}
```

---

### 漏桶算法（Leaky Bucket）

**原理：** 请求进入桶，以固定速率流出处理。超出桶容量则丢弃。平滑输出，不允许突发。

```
请求 → [桶] → 固定速率处理
桶满 → 丢弃
```

**对比：**
| 算法 | 突发流量 | 输出速率 | 适用场景 |
|------|---------|---------|---------|
| 令牌桶 | 允许 | 不固定 | API 限流 |
| 漏桶 | 不允许 | 固定 | 流量整形 |

---

### 滑动窗口计数器

**原理：** 统计滑动时间窗口内的请求数，超过阈值则拒绝。

```go
// Redis 实现滑动窗口限流
func isAllowed(ctx context.Context, rdb *redis.Client, key string, limit int, window time.Duration) bool {
    now := time.Now().UnixMilli()
    windowStart := now - window.Milliseconds()

    pipe := rdb.Pipeline()
    pipe.ZRemRangeByScore(ctx, key, "0", strconv.FormatInt(windowStart, 10))
    pipe.ZAdd(ctx, key, redis.Z{Score: float64(now), Member: now})
    pipe.ZCard(ctx, key)
    pipe.Expire(ctx, key, window)
    results, _ := pipe.Exec(ctx)

    count := results[2].(*redis.IntCmd).Val()
    return count <= int64(limit)
}
```

## 熔断（Circuit Breaker）

### 为什么需要熔断
- 下游服务故障时，快速失败，避免级联雪崩
- 给下游服务恢复时间
- 提升系统整体可用性

### 三种状态

```
Closed（正常）→ 错误率超阈值 → Open（熔断）
Open → 等待恢复时间 → Half-Open（探测）
Half-Open → 请求成功 → Closed
Half-Open → 请求失败 → Open
```

### Go 实现（使用 gobreaker）

```go
import "github.com/sony/gobreaker"

cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "order-service",
    MaxRequests: 3,                // Half-Open 状态最多允许 3 个请求
    Interval:    10 * time.Second, // Closed 状态统计窗口
    Timeout:     30 * time.Second, // Open 状态持续时间
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        // 失败率超过 60% 触发熔断
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return counts.Requests >= 10 && failureRatio >= 0.6
    },
})

result, err := cb.Execute(func() (interface{}, error) {
    return callDownstream()
})
if err == gobreaker.ErrOpenState {
    // 熔断中，快速失败或返回降级结果
    return fallback()
}
```

---

## 降级（Fallback）

熔断触发后的处理策略：
- 返回缓存数据
- 返回默认值
- 返回友好错误提示
- 调用备用服务

## 面试高频问题

**Q: 令牌桶和漏桶的区别？**
- 令牌桶：允许突发，桶里有令牌就能通过，适合 API 限流
- 漏桶：平滑输出，固定速率处理，适合流量整形

**Q: 熔断和限流的区别？**
- 限流：控制请求速率，保护自身
- 熔断：检测下游故障，快速失败，保护下游

**Q: 如何实现分布式限流？**
- Redis + Lua 脚本（原子操作）
- Redis 令牌桶：`redis-cell` 模块
- 网关层统一限流（Nginx、Kong）

**Q: 熔断器 Half-Open 状态的作用？**
- 探测下游是否恢复
- 允许少量请求通过，成功则关闭熔断，失败则继续熔断
- 避免下游刚恢复就被大量请求压垮

**Q: 如何设置合理的熔断阈值？**
- 错误率阈值：通常 50%-60%
- 最小请求数：避免样本太少误触发（如 10 个请求）
- 恢复时间：根据下游服务恢复速度设置（如 30s）
