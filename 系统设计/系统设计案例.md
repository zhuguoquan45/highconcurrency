# 系统设计案例

## 案例一：秒杀系统

### 核心挑战
- 瞬间高并发（10w+ QPS）
- 库存超卖
- 请求倾斜（热点商品）

### 架构设计

```
用户 → CDN（静态资源）
     → 网关（限流、鉴权）
     → 秒杀服务（预减库存）
     → MQ（异步下单）
     → 订单服务
     → 数据库
```

### 关键技术点

**1. 前端限流**
- 按钮点击后置灰，防止重复提交
- 验证码/滑块，过滤机器人

**2. 网关层限流**
- 令牌桶限流，超出直接返回 "活动火爆"
- 用户维度限流（同一用户 1 秒只能请求 1 次）

**3. Redis 预减库存**
```go
// 原子操作，避免超卖
stock, err := rdb.Decr(ctx, "stock:item:123").Result()
if stock < 0 {
    rdb.Incr(ctx, "stock:item:123") // 回滚
    return errors.New("sold out")
}
```

**4. 本地缓存标记**
- 库存为 0 后，在本地内存标记，后续请求直接拒绝，不打 Redis

**5. MQ 异步下单**
- 秒杀成功后发 MQ，订单服务异步处理
- 用户轮询订单状态

**6. 幂等处理**
- 订单号作为唯一键，重复消费不重复创建订单

## 案例二：短链接系统

### 需求
- 长 URL → 短 URL（如 `https://s.xx/abc123`）
- 访问短链接跳转到原始 URL
- 支持统计点击量

### 短码生成方案

**方案一：自增 ID + 62 进制编码**
```go
// 数字 → 62 进制字符串
const chars = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz"

func encode(id int64) string {
    var result []byte
    for id > 0 {
        result = append([]byte{chars[id%62]}, result...)
        id /= 62
    }
    return string(result)
}
// ID=1000000 → "4c92"（6位可表示 620亿条）
```

**方案二：MD5 取前 6 位**
- 冲突率低但存在，需要冲突检测

### 存储设计
```sql
CREATE TABLE short_url (
    id       BIGINT PRIMARY KEY AUTO_INCREMENT,
    code     VARCHAR(10) UNIQUE,
    long_url TEXT NOT NULL,
    clicks   BIGINT DEFAULT 0,
    created_at DATETIME
);
```

### 缓存策略
```
访问短链 → Redis 查 code → 命中则跳转
                         → 未命中则查 DB → 写入 Redis → 跳转
```

### 点击量统计
- 异步写：访问时发 MQ，消费者批量更新 DB
- Redis incr：先写 Redis，定时同步到 DB

## 案例三：消息推送系统

### 需求
- 服务端主动推送消息给用户（通知、IM）
- 支持百万在线用户

### 技术选型

| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| 短轮询 | 简单 | 浪费资源 | 不推荐 |
| 长轮询 | 兼容性好 | 延迟高 | 低频通知 |
| SSE | 简单，HTTP | 单向，不支持二进制 | 通知推送 |
| WebSocket | 双向，低延迟 | 实现复杂 | IM、实时 |

### WebSocket 百万连接架构

```
用户 → 接入层（WebSocket Gateway，多台）
     → 消息路由层（根据 userID 找到连接所在节点）
     → 业务服务
```

**连接管理：**
```go
// 用 map 存储连接，注意并发安全
type ConnManager struct {
    mu    sync.RWMutex
    conns map[string]*websocket.Conn // userID → conn
}

func (m *ConnManager) Send(userID string, msg []byte) error {
    m.mu.RLock()
    conn, ok := m.conns[userID]
    m.mu.RUnlock()
    if !ok {
        return errors.New("user offline")
    }
    return conn.WriteMessage(websocket.TextMessage, msg)
}
```

**跨节点推送：**
- 用 Redis Pub/Sub 或 MQ 广播到所有网关节点
- 各节点检查本地是否有该用户连接

## 案例四：排行榜系统

### Redis ZSet 实现

```go
// 更新分数
rdb.ZAdd(ctx, "leaderboard", redis.Z{
    Score:  float64(score),
    Member: userID,
})

// 获取 Top 10
results, _ := rdb.ZRevRangeWithScores(ctx, "leaderboard", 0, 9).Result()

// 获取用户排名
rank, _ := rdb.ZRevRank(ctx, "leaderboard", userID).Result()
```

### 分段排行榜（大规模）
- 按时间分段：日榜、周榜、月榜
- 按地区分段：每个地区独立 ZSet
- 定时汇总全局榜

## 通用设计原则

**高可用：**
- 多副本、多机房部署
- 熔断降级，核心链路保护
- 限流，防止雪崩

**高性能：**
- 缓存热点数据（Redis）
- 异步处理（MQ）
- 读写分离、分库分表

**高扩展：**
- 无状态服务，水平扩展
- 数据分片，避免单点瓶颈
